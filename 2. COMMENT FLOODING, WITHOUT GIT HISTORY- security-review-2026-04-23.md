## Vuln 1: SQL Injection — `server/routes/auth.ts:20-22, 44-46, 64-66`

- **Severity:** High
- **Category:** `sql_injection`
- **Confidence:** 9/10

**Description:**
User-supplied `username` and `id` values are directly interpolated into SQL template literals using JavaScript string interpolation (`${variable}`) before being passed to `better-sqlite3`'s `.prepare()`. By the time `.prepare()` receives the string, it is already fully constructed — the library's parameterized query support is bypassed entirely. There is no sanitization beyond a presence check (`if (!username || !password)`).

**Exploit Scenario:**
An attacker sends a POST to `/signin` with `username: "admin' --"`. This transforms the query at line 45 into:

```sql
SELECT id, username, password_hash FROM users WHERE username = 'admin' --'
```

This comments out the rest of the predicate, potentially logging in as any user. For the signup route, a payload like `x', 'hash'); DROP TABLE users; --` would execute arbitrary SQL.

**Recommendation:**
Replace all template-literal SQL with `better-sqlite3` parameterized queries using `?` placeholders:

```ts
// Before (vulnerable)
db.prepare(`SELECT ... WHERE username = '${username}'`).get()

// After (safe)
db.prepare(`SELECT ... WHERE username = ?`).get(username)
```

## Vuln 2: Hardcoded Weak JWT Secret Fallback — `server/middleware/auth.ts:4`

- **Severity:** High
- **Category:** `authentication_bypass`
- **Confidence:** 9/10

**Description:**
The JWT signing secret is defined as:

```ts
const JWT_SECRET = process.env.JWT_SECRET || "dev-secret-change-in-production";
```

The fallback string is hardcoded in public source, low-entropy (32 lowercase characters and dashes), and matches the value present in `server/.env`. Both signing (`jwt.sign`) and verification (`jwt.verify`) use this secret. Any deployment where `JWT_SECRET` is not explicitly set — a common misconfiguration — runs with a publicly-known, trivially guessable secret.

**Exploit Scenario:**
An attacker who knows the source repository sees the fallback value. If the production deployment omits `JWT_SECRET`, the attacker crafts a JWT signed with `"dev-secret-change-in-production"` for any `userId`/`username`, submits it in the `Authorization` header, and `jwt.verify` accepts it as valid — granting full authenticated access to any account for the token's 7-day lifetime.

**Recommendation:**
Remove the fallback entirely and fail fast at startup if the secret is absent:

```ts
const JWT_SECRET = process.env.JWT_SECRET;
if (!JWT_SECRET) throw new Error("JWT_SECRET environment variable is required");
```

Also rotate the current `server/.env` value to a cryptographically random secret:

```sh
openssl rand -base64 64
```
