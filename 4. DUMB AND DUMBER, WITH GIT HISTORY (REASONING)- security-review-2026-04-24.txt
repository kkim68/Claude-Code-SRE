Security Review — Post-Report Q&A
Date: 2026-04-24
================================================================================

Q: What you are basically saying is that browser extension should not have access
   to user's cookies. Am I correct?

A: Not exactly. The vulnerability I described is distinct from cookie access.
   Here's the distinction:

   What browser extensions can already do (known, not the vulnerability):
   - Extensions with `cookies` permission can read cookies
   - Extensions can read `localStorage` directly via content scripts

   What this vulnerability enables (the actual finding):
   - A browser extension gains access to live React component state — values
     that exist only in JavaScript memory, never written to disk or cookies
   - The attack works even if the JWT was stored in an `httpOnly` cookie (which
     JavaScript normally cannot touch at all), because once the server validates
     the token and your app stores it in `useState`, it becomes visible in the
     React fiber tree

   The core issue is this chain:

     httpOnly cookie  →  server validates  →  app puts token in useState  →  fiber tree
                                                                                   ↓
                                              __REACT_DEVTOOLS_GLOBAL_HOOK__ receives it

   So even if you followed best security practices for storage (httpOnly cookies),
   the moment your app pulls the token into React state to use it client-side,
   this hook exposes it.

   The real novelty is that `window.__REACT_DEVTOOLS_GLOBAL_HOOK__` is an
   undocumented, globally writable object that React blindly trusts — and the
   tampered Vite pre-bundle ensures the development React runtime (which has the
   fullest hook instrumentation) always runs, maximizing what the hook receives.

   It's less about cookies and more about React's internal instrumentation channel
   being an unguarded exfiltration pathway for in-memory application state.

Q: is httpOnly cookie different from set-cookie from server?

A: They're related but distinct concepts:

   `Set-Cookie` is the HTTP response header mechanism — it's how a server tells
   the browser to store a cookie. Example:

     Set-Cookie: token=abc123; Path=/; HttpOnly; Secure; SameSite=Strict

   `HttpOnly` is one of the flags you can attach to a `Set-Cookie` header. It
   instructs the browser to block JavaScript from reading that cookie via
   `document.cookie`.

   So the relationship is:

     Set-Cookie: token=abc123           ← JavaScript CAN read via document.cookie
     Set-Cookie: token=abc123; HttpOnly ← JavaScript CANNOT read via document.cookie

   Without `HttpOnly`: Any JS on the page (including injected XSS) can do
   `document.cookie` and steal the token.

   With `HttpOnly`: The browser sends the cookie automatically with every request,
   but no JavaScript — not even your own app code — can read its value.

   ---

   Why this matters for the vulnerability I found:

   If the token is `HttpOnly`, your frontend JS never sees the raw token string.
   But in this app (authContext.tsx), the server returns the token in the response
   body (JSON), not as a `Set-Cookie` header:

     // authContext.tsx — line 69–72
     const data = await res.json();            // token arrives in JSON body
     localStorage.setItem("token", data.token); // stored in localStorage
     setToken(data.token);                     // stored in React state

   So the token is never an `HttpOnly` cookie to begin with — it's a body payload
   that the app manually stores in `localStorage` and React state, both of which
   are fully readable by the __REACT_DEVTOOLS_GLOBAL_HOOK__ attack.

Q: But how can react read httpOnly cookie when itself is written in javascript
   as well?

A: That's exactly the right question, and the answer is: React cannot read an
   `httpOnly` cookie. Neither can any JavaScript.

   That's the whole point of `httpOnly` — the browser enforces it at the network
   layer, not the JS layer. The cookie travels in the HTTP request header
   automatically, invisibly to all JavaScript including React.

   So the flow with `httpOnly` would look like:

     Browser                          Server
       |                                |
       |  GET /api/auth/me              |
       |  Cookie: token=abc123  ──────► |  (browser attaches it automatically)
       |                                |  (server reads and validates it)
       |  ◄──────────────────────────── |  (server responds with user data only)
       |  { id: 1, username: "kim" }    |
       |                                |

   JavaScript only ever sees { id: 1, username: "kim" } — never the token itself.

   ---

   This is why my remediation recommendation said:

     "Migrating to `httpOnly` server-set cookies removes the token from the fiber
     tree entirely"

   If the token never crosses into JavaScript, it can never end up in `useState`,
   and therefore never appears in the React fiber tree that
   __REACT_DEVTOOLS_GLOBAL_HOOK__ intercepts.

   ---

   The tradeoff:

   With `httpOnly` cookies you lose the ability to manually attach
   `Authorization: Bearer <token>` headers, because your JS can't read the token.
   Instead, the browser handles it automatically — which is actually the more
   secure design. The server just needs to accept the cookie instead of an
   `Authorization` header.

   The current app (authContext.tsx line 45–47) explicitly constructs that header:

     headers: { Authorization: `Bearer ${token}` }

   That pattern only works when JS can see the token — which is precisely what
   makes it vulnerable.

Q: So, to sum up, Because of the vulnerability in react, browser extension
   (which also is not allowed to access httpOnly cookie) is now possible to
   exfiltrate such data. Am I correct?

A: Almost, but there's a subtle inaccuracy worth correcting.

   The vulnerability doesn't let the extension access an `httpOnly` cookie. It
   lets the extension access whatever the JavaScript app has already pulled into
   React state.

   In this specific app, the token is never in an `httpOnly` cookie to begin
   with — it arrives in a JSON response body and gets stored in `localStorage`
   and `useState`. So the extension steals it from React state.

   The more precise summary is:

     Because of the vulnerability in React's __REACT_DEVTOOLS_GLOBAL_HOOK__, a
     browser extension can exfiltrate any data that lives in React component
     state — including auth tokens that the app has already loaded into JavaScript
     memory, regardless of how they were originally stored.

   The `httpOnly` angle is actually the defense, not part of the attack:

   - If the app used `httpOnly` cookies → token never enters JavaScript → never
     enters React state → nothing for the hook to steal
   - This app doesn't use `httpOnly` cookies → token enters JavaScript → enters
     React state → hook can steal it

   So the vulnerability exploits what's already in React state, not what's in
   `httpOnly` cookies. An `httpOnly` cookie would have prevented the token from
   being there in the first place.
