### Candidate 1: Command Injection in `/api/health` via `service` Query Parameter

- **File:** `server/index.ts` (health check route)
- **Category:** Command Injection / Input Validation
- **Initial Severity Assessment:** HIGH
- **Final Confidence:** 1/10 — **Excluded (False Positive)**

#### What Was Flagged

The health check endpoint reads a `service` query parameter from the request and passes it to `execFile()`:

```typescript
const service = (req.query.service as string) || "api.themoviedb.org";
execFile('ping', ['-c', '1', '-W', '2', service], (err, stdout) => {
  // ...
});
```

The concern was that a user-controlled string is used in a system subprocess call without any allow-list or sanitization, potentially enabling:
- Shell metacharacter injection (e.g., `; whoami`, `&& cat /etc/passwd`)
- Ping flag injection (e.g., appending `-R` or `-p <payload>` to the hostname)

#### Why It Was Excluded

The critical distinction is between `exec()` and `execFile()`. Node.js `exec()` passes its argument to `/bin/sh -c`, which means the shell tokenizes the string and interprets metacharacters. `execFile()`, by contrast, **does not spawn a shell at all**. It passes each array element as a discrete, literal argument directly to the target binary via `execve()`.

Concretely, when the server receives `GET /api/health?service=api.themoviedb.org%20-R`, the OS-level call becomes:

```
execve('/bin/ping', ['ping', '-c', '1', '-W', '2', 'api.themoviedb.org -R'], envp)
```

The `ping` binary receives `api.themoviedb.org -R` as **one indivisible hostname string**, not two arguments. It has no mechanism to split this string into a hostname and a flag — that tokenization only happens in a shell. Ping will attempt to DNS-resolve the string `"api.themoviedb.org -R"`, fail, and return an error. No additional flags execute.

Similarly, `; whoami` or `&& cat /etc/passwd` appended to the service parameter are also treated as a single hostname string. There is no shell to interpret the separator characters.

**The prior commit that introduced `execFile()` over `exec()` was precisely the correct fix.** The code is now secure for this attack surface.

### Candidate 2: Sensitive Data Exposure via Winston Logging of `service` Parameter

- **File:** `server/index.ts` (health check route, logger calls)
- **Category:** Data Exposure / Information Disclosure
- **Initial Severity Assessment:** MEDIUM
- **Final Confidence:** 1/10 — **Excluded (False Positive)**

#### What Was Flagged

The newly added Winston logger records the user-supplied `service` parameter on every health check request:

```typescript
logger.log('info', 'Health Check Requested:');
logger.log('info', service);
```

The concern was that logging unsanitized user input could enable an attacker to:
- Inject content into log streams to mislead operators (log spoofing)
- Reveal internal service names or IPs via reconnaissance by observing what gets logged
- Cause log data to accumulate that could be exfiltrated if the logging backend is compromised

#### Why It Was Excluded

The `service` parameter is a **hostname or domain name** — the same kind of data as a URL. The applicable filtering rules explicitly state: *"Logging URLs is assumed to be safe"* and *"Logging non-PII data is not a vulnerability even if the data may be sensitive. Only report logging vulnerabilities if they expose sensitive information such as secrets, passwords, or personally identifiable information (PII)."*

A hostname contains none of those categories. Even if an attacker passes a crafted hostname like `attacker.com` or `127.0.0.1`, logging that string does not expose credentials, PII, or application secrets. It is also explicitly excluded under the log spoofing rule: *"Outputting un-sanitized user input to logs is not a vulnerability."*

There is no concrete exploit path here — no attacker gains meaningful advantage from seeing their own input echoed in server logs.

### Candidate 3: Supply Chain Risk — Custom Forked Winston Package (`kkim68/winston`)

- **File:** `server/package.json`, `server/package-lock.json`
- **Category:** Supply Chain Security
- **Initial Severity Assessment:** MEDIUM
- **Final Confidence:** 2/10 — **Excluded (False Positive)**

#### What Was Flagged

Instead of the official `winston` npm package, `server/package.json` depends on a GitHub fork:

```json
"winston": "kkim68/winston"
```

The lockfile resolves this to:

```
git+ssh://git@github.com/kkim68/winston.git#55cdb50deb9eb93ff65afc4678b60b566181cead
```

The fork also introduces a nested custom dependency, `vllogger` (from `kkim68/vllogger`), not present in the official Winston package. The concern was:

- The fork contents are not vetted by the public npm security pipeline
- A malicious actor who gains access to the fork repository could inject a backdoor
- That backdoor would be pulled into every deployment

#### Why It Was Excluded

The repository owner is `kkim68` — the same GitHub identity that owns both `kkim68/winston` and `kkim68/vllogger`. The developer is depending on **their own fork** of an upstream package, which is a normal and common development pattern (e.g., for applying a patch that hasn't been merged upstream, or adding a custom transport).

For this to be a genuine supply chain attack, a third-party adversary would need to compromise the `kkim68` GitHub account — at which point they would also have access to this repository directly, making the supply chain vector redundant. This is not an inherent weakness of the code; it is an account-security concern for the developer, not a code-level vulnerability.

The finding was also noted to fall under the "lack of hardening measures" exclusion: *"Code is not expected to implement all security best practices, only flag concrete vulnerabilities."* Preferring the official npm package over a personal fork is a best practice, not a mandatory security requirement with a proven attack path.

**Recommendation (non-blocking):** Consider switching to the official `winston` npm package if the fork customizations can be upstreamed or are no longer needed. If the fork is intentional, document the reason in a comment in `package.json`.

### Candidate 4: Weak Winston Logger Configuration

- **File:** `server/index.ts` (logger initialization, lines ~11–17)
- **Category:** Data Exposure / Improper Configuration
- **Initial Severity Assessment:** MEDIUM
- **Initial Confidence from Phase 1:** 6/10 — **Excluded before filtering (below threshold)**

#### What Was Flagged

The logger is configured with `format.simple()` and colorized console output, and is assigned directly to `module.exports`:

```typescript
const logger = module.exports = winston.createLogger({
  transports: [new winston.transports.Console()],
  format: winston.format.combine(
    winston.format.colorize({ all: true }),
    winston.format.simple()
  )
});
```

Concerns raised:
- `module.exports = ` assignment is non-standard and could cause the logger to be imported unintentionally by other modules
- No log-level filtering or sensitive-field masking is configured
- Console transport may send unfiltered logs to centralized log aggregation systems (Docker/Kubernetes)

#### Why It Was Excluded

This finding scored only 6/10 confidence in Phase 1 and was excluded before the individual false-positive filter was applied. The issues are stylistic and configuration-quality concerns rather than concrete, exploitable vulnerabilities:

- `module.exports = ` is unusual but does not expose an attack surface on its own — it is the module system working as designed
- Lack of log-level filtering is a defense-in-depth gap, not a vulnerability; the filtering rules state code is not expected to implement all hardening measures
- Centralized log exfiltration requires an attacker to have already compromised the logging infrastructure, which is a separate threat
