Claude Code assumes a Git repository as its scanning foundation. In other words, Claude Code 
primarily scans files that have been newly added or modified in Git. This is a deliberate and 
rational design choice, driven by the need to balance scan efficiency with preserving the user's 
context window. To force a full scan without relying on Git history, follow the steps below.

# 1. Clone the project folder (Copy & Paste)

# 2. Remove the ".git" folder inside the cloned project folder

# 3. Git initialization
  
  git init

# 4. Create a bare repo (location does not matter)
  
  git init --bare /tmp/fake-origin

# 5. Connect a "local temporary" remote to the current project
  
  git remote add origin /tmp/fake-origin

# 6. Git commit
  
  git add *
  git commit -m "."

# 7. Push to create origin/HEAD
  
  git push origin main 

# 8. Set head
  
  git remote set-head origin main

# 9. Run Claude Code /security-review

Note that even after following these steps, Claude Code may still attempt to scan only 
"changed files." When this happens, explicitly include "Do a Full Scan" in your prompt.

This is necessary because of specific threat scenarios we discovered. Take Triple Obfuscation as an example. When /security-review is run against an obfuscated codebase, it tends to reference the pre-obfuscation version of the source through Git history, and performs its vulnerability scan against that original baseline. In a standard development workflow, this behavior is arguably reasonable. However, it breaks down in a scenario where a scanning party extracts source code from a deployed, web-based application and runs a security scan against it. In that context, the "original source" is simply not accessible. An attacker could exploit this gap by planting a vulnerability, obfuscating the code, and shipping it to production. The scanner, unable to reach the clean baseline, would miss the vulnerability entirely. In this scenario, the party conducting the scan has no access to Git history whatsoever, leaving a full scan as the only viable option.

------------------------------------------------------------------------

It appears that /security-review references locally cached "previous scan history" when 
it runs low on context. This cache is located under ~/.claude/projects/. Clearing the 
relevant entries is necessary to ensure consistent and reproducible experiments.

This cache is not cleared by /clear or /reset commands, so it must be manually removed 
before each experiment.
